# Part 04 Spec: 运行时状态与存档层

## 1. 目标

让游戏在单主线下稳定维护当前节点、轮次、隐藏态度值、历史摘要和设置，并支持中断后恢复。

## 2. 范围

1. 主线当前节点状态。
2. 当前轮次索引。
3. 隐藏态度值的更新规则。
4. 历史摘要与已读状态。
5. 存档读写与恢复。
6. 基础设置项，如 BGM 开关。

## 3. 明确不做

1. 不做 AI 生成。
2. 不做知识编译。
3. 不做正式视觉表现。
4. 不决定按钮具体文案。

## 4. 依赖与前置

1. 依赖 Part 01 的桌面文件系统能力。
2. 依赖 Part 02 的 story node 契约。
3. 为 Part 05 和 Part 06 提供运行时状态。

## 5. 契约与边界

1. 运行时状态必须是可序列化的。
2. 态度值只影响风格，不影响主线走向。
3. 存档格式必须能兼容最小版本演进。
4. 状态层只负责事实状态，不负责视图排版。

## 6. 关键流程

1. 初始化存档或创建默认状态。
2. 读取当前 story node。
3. 接收玩家的附和或反驳选择。
4. 更新态度值与轮次。
5. 推进到下一主线节点或下一轮。
6. 持久化存档。

## 7. 错误处理要求

1. 存档不存在时自动回退到初始状态。
2. 存档损坏时要能给出恢复策略或重新开始提示。
3. 节点配置丢失时，不能静默推进到未知状态。

## 8. 测试策略

黑盒测试：

1. 新玩家能正常开始。
2. 已有存档能恢复到之前进度。
3. 玩家选择不同态度后，主线仍按预定推进。

白盒测试：

1. 态度值增减与边界钳制正确。
2. 序列化与反序列化字段一致。
3. 节点推进逻辑无断链。

## 9. 验收条件

1. 存档可创建、覆盖、读取。
2. 隐藏态度值有明确上下界和更新规则。
3. 玩家中断后可继续游玩。
4. 任何一次选择都不会把主线推进成分支剧情。

## 10. 交付物

1. 运行时状态模型。
2. 存档仓储模块。
3. 态度值引擎。
4. 状态层测试。

## 11. 执行状态

Status: completed

Audit:

1. See `docs/audits/2026-04-23-part-01-04-audit.md`

已完成：

1. 已建立可序列化的运行时状态模型，包含当前节点、轮次、态度值、历史摘要、已读节点与基础设置。
2. 已建立最小态度值引擎，支持附和与反驳选择、上下界钳制，以及终点节点的单主线回合推进。
3. 已建立文件存档仓储，支持默认存档创建、存档覆盖读取与损坏文件回退。
4. 已完成黑盒与白盒最小验证，覆盖新玩家开始、已有存档恢复、损坏存档重建、态度值边界与序列化往返。

明确延期：

1. 暂不接入 Electron IPC、正式设置页与真实用户数据目录策略，本部分先保证纯状态模型和仓储行为稳定。
2. 暂不实现多存档槽位、迁移器链式升级与复杂历史摘要拼装。

## 12. 风险与回滚边界

1. 若状态层混入提示词或 UI 文案，会导致 Part 05 与 Part 06 难以维护。
2. 若存档结构未提前收敛，后续版本兼容会变得脆弱。
