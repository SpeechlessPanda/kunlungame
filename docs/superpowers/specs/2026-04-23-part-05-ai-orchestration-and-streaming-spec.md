## 1. 目标

将当前节点状态、检索结果、风格状态和反剧透约束组装成稳定的模型请求，并把响应以流式方式交给上层消费。

## 2. 范围

1. 提示词组装。
2. OpenAI 兼容接口接入。
3. 流式文本消费。
4. 两个动态选项文案生成。
5. 失败提示与本轮重试。

## 3. 明确不做

1. 不负责 Markdown 编译。
2. 不负责状态持久化本身。
3. 不负责背景图渲染。
4. 不负责真实素材加载。

## 4. 依赖与前置

1. 依赖 Part 03 的检索结果。
2. 依赖 Part 04 的当前状态。
3. 为 Part 06 和 Part 08 提供对话流输出。

## 5. 契约与边界

1. 输入至少包括当前节点、态度值、知识片段。
2. 输出至少包括文本流片段、选项结果、完成事件、错误事件。
3. 模型层不得自行推进主线节点。
4. 生成的两个选项必须分别映射附和型与反驳型语义。
5. prompt 必须显式包含中文输出约束和未来主题禁区。

## 6. 关键流程

1. 读取当前状态。
2. 拉取当前节点相关知识片段。
3. 组装系统与用户提示内容。
4. 调用对话编排 facade，先构建 prompt，再委托底层流式实现。
5. 将响应按流式片段向上游转发。
6. 在结束时输出两个动态选项。

## 7. 错误处理要求

1. 模型不可用时，返回清晰错误状态。
2. 流中断时，允许重试当前轮。
3. 检索为空时，启用受控降级提示词，而不是自由发散。
4. 任何异常都不能把未完成的输出伪装成成功完成。

## 8. 测试策略

黑盒测试：

1. 模型可用时能逐步输出文本。
2. 每轮都能得到两个上下文相关选项。
3. 模型失败时可重试。

白盒测试：

1. 提示词中包含节点约束、风格约束、反剧透约束。
2. 流事件序列顺序正确。
3. 动态选项映射到内部态度枚举正确。
4. `complete` 与 `error` 事件不能混用或倒序。

## 9. 验收条件

1. 对话文本不是整段一次性出现。
2. 选项文案与当前回复语境匹配。
3. 模型错误时 UI 可获取明确失败信号。
4. 生成内容不会越过当前主线节点剧透后续核心内容。

## 10. 交付物

1. Prompt builder。
2. 对话编排 facade。
3. 后续真实模型适配客户端。
4. AI 层单元测试与失败路径测试。

## 11. 风险与回滚边界

1. 若提示词里没有明确的边界约束，单主线设定会被模型打穿。
2. 若流事件和状态更新混在一起，会导致调试困难和 UI 不稳定。
