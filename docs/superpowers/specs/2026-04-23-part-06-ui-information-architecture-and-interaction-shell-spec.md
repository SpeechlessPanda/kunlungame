# Part 06 Spec: UI 信息架构与交互壳

## 1. 目标

定义游戏 UI 的信息结构、交互区域和组件边界，让渲染层先具备稳定的对话容器、按钮结构和状态展示骨架。

## 2. 范围

1. 页面信息架构。
2. 对话区域、选项区域、状态区域、设置入口的布局边界。
3. 基础组件划分与交互壳。
4. 占位态、加载态、错误态的 UI 结构。

## 3. 明确不做

1. 不实现正式美术风格定稿。
2. 不接入真实背景素材文件。
3. 不改写 AI 或状态层逻辑。

## 4. 依赖与前置

1. 依赖 Part 01 的渲染工程。
2. 依赖 Part 04 和 Part 05 的状态与流事件契约。
3. 为 Part 07 提供可挂载的表现层容器。

## 5. 契约与边界

1. UI 壳层只消费状态，不生成业务事实。
2. 组件边界至少拆为 `GameShell`、`DialogPanel`、`ChoiceButtons`、设置或状态显示组件。
3. 每个组件要有明确输入属性与输出事件。

## 6. 关键流程

1. 页面初始化展示空态或加载态。
2. 收到文本流后逐步刷新对话面板。
3. 收到选项后渲染两个按钮。
4. 玩家选择后进入等待态。
5. 出错时展示错误态与重试入口。

## 7. 错误处理要求

1. 加载中、无数据、错误态都要有清楚视觉区分。
2. 未收到选项时按钮区域不能显示脏状态。
3. 组件 prop 缺失时，要有空态渲染而不是崩溃。

## 8. 测试策略

黑盒测试：

1. 页面能展示文本区和按钮区。
2. 收到流片段后文本逐步更新。
3. 错误发生时出现重试入口。

白盒测试：

1. 组件事件触发正确。
2. 加载态与错误态条件分支正确。
3. 对话文本缓存与可见文本计算逻辑正确。

## 9. 验收条件

1. 至少有一个稳定可测试的游戏主界面骨架。
2. 对话区、选项区、状态区的职责清晰。
3. UI 不依赖真实素材即可运行。
4. 页面结构能支持 Part 07 接入视觉表现。

## 10. 交付物

1. 主界面骨架组件。
2. 基础交互组件。
3. 加载态、错误态、空态。
4. UI 层组件测试。

## 11. 风险与回滚边界

1. 若本部分直接掺入视觉定稿，会导致后续素材替换困难。
2. 若组件边界不清晰，Part 07 和 Part 08 会不断返工。

## 12. 实施进度（feat/ui-shell）

已完成：

1. 信息架构：`GameShell` 组合 `StatusBar`、`DialogPanel`、`ChoicePanel`、`SettingsPanel`，背景/角色资源位分层明确。
2. 交互壳：空态 / 加载态 / 流式态 / 等待选择态 / 错误态的 UI 路径齐全；错误态提供“重试”入口。
3. 文本流：`DialogueStreamBuffer` + `createTurnController` 组合完成文本缓存、逐字显示、可跳过与重置。
4. 状态机：`reduceUiTurn` 封装合法事件迁移，组件只消费快照，不生成业务事实。
5. 测试：`uiStateMachine`、`dialogueStreamBuffer` 白盒测试；`tests/e2e/rendererShell.spec.ts` 覆盖空态、流式文本、选项渲染、错误重试。

有意延后：

1. 与真实 AI 流适配器的对接（等待 Part 05 完成）。
2. 选择落盘与态度联动（等待 Part 04 Save 层联通）。
3. 键盘导航与焦点环细节（等待真实键位方案）。
