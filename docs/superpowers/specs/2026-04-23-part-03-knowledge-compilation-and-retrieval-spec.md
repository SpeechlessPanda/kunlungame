# Part 03 Spec: 知识库编译与检索层

## 1. 目标

建立从 Markdown 原始内容到结构化知识数据的编译链路，并提供面向主线节点的稳定检索能力。

## 2. 范围

1. Markdown 内容解析。
2. 结构化知识 JSON 生成。
3. 基于节点、关键词、主题的检索。
4. 检索结果数量限制与排序策略。
5. 检索为空时的降级策略。

## 3. 明确不做

1. 不负责主线推进。
2. 不直接调用 AI。
3. 不负责 UI 展示。
4. 不负责真实内容的最终质量审核。

## 4. 依赖与前置

1. 依赖 Part 02 的内容契约。
2. 为 Part 05 提供检索输入。

## 5. 契约与边界

1. 输入是受约束的 Markdown 原文。
2. 输出是稳定结构化条目数组。
3. 检索器只关心数据匹配，不关心生成语气。
4. 排序策略优先 story node 显式命中，其次关键词命中。

## 6. 关键流程

1. 读取 Markdown。
2. 拆分条目。
3. 解析元信息与正文。
4. 输出结构化文件。
5. 在运行时按当前节点与关键词筛选候选知识片段。

## 7. 错误处理要求

1. Markdown 格式错误时输出明确定位信息。
2. 编译结果为空时阻止继续进入正式检索流程。
3. 检索为空时返回降级结果，而不是抛未处理异常。

## 8. 测试策略

黑盒测试：

1. 给定符合格式的 Markdown，能生成结构化 JSON。
2. 给定节点和关键词，能返回预期知识条目。
3. 检索为空时，返回降级结果。

白盒测试：

1. 解析器对元信息和正文分段处理正确。
2. 检索排序规则稳定。
3. 条目过滤上限和截断策略一致。

## 9. 验收条件

1. 存在可重复执行的知识编译脚本。
2. 编译结果能被版本化管理或稳定生成。
3. 检索层能按节点返回有限且相关的结果。
4. 编译错误与检索空结果都有清晰处理路径。

## 10. 交付物

1. 编译脚本。
2. 编译后的知识文件。
3. 检索服务模块。
4. 内容格式验证测试。

## 11. 风险与回滚边界

1. 若 Markdown 规则过松，后续 AI 生成会不断吃到脏数据。
2. 若检索器承担过多生成职责，会和 Part 05 耦合。
