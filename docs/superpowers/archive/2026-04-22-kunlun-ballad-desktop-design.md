# 昆仑谣文化叙事游戏设计文档

**日期：** 2026-04-22

## 1. 项目目标

本项目首版是一个 Windows 桌面叙事游戏。玩家沿单一主线推进，在与 AI 的多轮对话中逐步理解《昆仑谣》歌词背后的中国文化脉络，以及从歌词主题延展出的相关文化内容。

首版的核心目标如下：

1. 提供一条稳定、可持续推进的主线，不做分支剧情树。
2. 通过对话推进知识理解，而不是通过百科式堆砌文本直接灌输信息。
3. 接入真实本地部署模型，支持基于知识检索的流式回答。
4. 让玩家的每轮选择影响 AI 的说话风格，但不改变主线事实推进。
5. 首轮完整体验目标不少于 5 分钟，并支持保存进度后继续游玩。

## 2. 产品范围

首版范围包含以下内容：

1. Windows 桌面客户端。
2. 单主线章节推进机制。
3. 基于 Markdown 内容整理的知识库导入与结构化落地。
4. OpenAI 兼容接口的本地模型接入。
5. AI 流式输出。
6. 每轮两个动态生成的上下文选项。
7. 隐藏态度值系统，用于累积 AI 风格倾向。
8. 自动或显式的进度保存能力。
9. 基础 UI、背景切换和 BGM 控制。

首版暂不要求以下内容：

1. 多分支剧情。
2. 配音系统。
3. 完整的后台内容管理系统。
4. 大规模开放式自由输入交互。

## 3. 核心体验定义

玩家进入游戏后，系统加载当前主线节点，展示对应场景与对话。AI 不自由发散到整条文化史，而是被约束在当前节点可讲述的内容范围内。每一轮对话结束后，系统根据当前语境生成两个具体选项：

1. 一个表达附和、顺从、认同当前话题的选项。
2. 一个表达反驳、顶嘴、质疑或轻微对抗态度的选项。

玩家点击其中之一后：

1. 主线仍按预定节点推进。
2. 系统记录本轮态度取向。
3. 隐藏态度值发生变化。
4. AI 后续输出的措辞、口吻、亲密度、夸赞感或傲娇感发生变化。

因此，玩家影响的是陪伴感和互动风格，而不是故事分叉方向。

## 4. 系统架构

建议首版按以下 5 层拆分：

### 4.1 桌面壳层

负责应用窗口、静态资源加载、本地文件读写、设置项和存档管理。

### 4.2 剧情状态层

负责维护主线节点、当前段落、玩家历史选择、隐藏态度值、已读状态和当前存档进度。

### 4.3 知识检索层

负责从用户整理的 Markdown 内容中抽取结构化知识条目，并在每次生成前根据当前主线节点、主题标签和检索关键词选出当前轮最相关的知识片段。

### 4.4 AI 编排层

负责把主线进度、知识检索结果、风格状态、输出长度约束、禁止剧透约束和选项生成要求组装为模型请求，并通过 OpenAI 兼容接口完成流式生成。

### 4.5 表现层

负责对话框、角色区域、背景切换、文字逐步出现、选项按钮、加载状态、错误提示和 BGM 控制。

## 5. 主线设计约束

本轮设计不直接写死主线具体章节内容。主线具体文本、节点顺序和文化内容铺陈将由用户后续继续确定。

当前阶段只定义主线必须满足的结构约束：

1. 主线是单线推进结构。
2. 主线由多个可配置节点组成。
3. 每个节点至少包含以下信息：
   1. 节点标识。
   2. 剧情目标。
   3. 对应文化主题。
   4. 可检索关键词。
   5. 建议背景或场景线索。
   6. 建议语气范围。
   7. 下一节点入口。
4. 节点数量与长度配置应能够支撑首轮游玩时长达到或超过 5 分钟。

后续在真正制作主线内容时，可继续根据《昆仑谣》歌词核心主题逐步向中国神话、礼乐文明、诸子百家、历史文脉与现代转化等方向扩展，但这些具体内容不在本版设计文档中写死。

## 6. 知识库设计

知识库原始来源由用户后续上网整理，并保存为 Markdown 文档。系统需要提供从 Markdown 到结构化知识条目的内容处理路径。

每条结构化知识至少包含：

1. 主题。
2. 出处。
3. 摘要。
4. 延伸内容。
5. 适配的主线节点。
6. 可检索关键词。

必要时可追加以下字段：

1. 相关歌词片段。
2. 朝代或历史阶段标签。
3. 文化母题标签。
4. 推荐语气。
5. 推荐背景图标签。

知识库的设计原则如下：

1. 先服务主线节点，不追求百科全书式完备。
2. 每条内容应能被当前节点稳定调用。
3. 支持未来继续扩容，而不需要重做整体结构。

## 7. AI 生成与约束

AI 每轮生成必须遵守以下规则：

1. 只能围绕当前主线节点发言。
2. 生成前必须先检索知识库，再组织回答。
3. 不允许提前讲穿后续节点的核心内容。
4. 输出必须支持流式显示。
5. 每轮都要生成两个符合当前上下文的具体选项文案，而不是固定按钮标签。
6. 两个选项分别承载附和型态度与反驳型态度。
7. AI 风格必须受隐藏态度值影响，但事实内容与主线推进不能被态度值改写。

推荐的生成过程如下：

1. 读取当前节点状态。
2. 检索与当前节点相关的知识片段。
3. 结合隐藏态度值确定当前说话风格强度。
4. 生成本轮流式回复。
5. 生成两条与上下文匹配的选项文案。
6. 等待玩家选择后更新态度值与主线状态。

## 8. 风格系统

玩家每轮选择不影响剧情事实，但会影响 AI 的人格表现。

风格系统至少需要支持：

1. 一个可累积的隐藏态度值。
2. 态度值向“可爱、夸赞、亲近”方向偏移时的风格模板。
3. 态度值向“傲娇、顶嘴、带一点别扭感”方向偏移时的风格模板。
4. 风格变化幅度控制，避免语气突然跳变。

风格系统的目标不是创造第二条剧情，而是增强重复对话中的陪伴感和角色感。

## 9. 表现层要求

首版表现层不以复杂演出为先，但需要预留足够空间接入后续资源。

已确认的表现方向如下：

1. 后续会接入用户提供的角色形象与背景图。
2. 背景图数量应支持随着主线推进自然变化和转场。
3. 对话不需要配音。
4. 《昆仑谣》可以作为低干扰的 BGM 使用，不能喧宾夺主。

因此，首版 UI 需要先完成以下能力：

1. 背景切换机制。
2. 角色区域占位或可替换资源位。
3. BGM 播放与基础音量控制。
4. 可承载长对话的文字区域。
5. 两个动态选项按钮。

## 10. 存档与继续游玩

首版需要支持保存进度，以便玩家中断后继续。

至少需要持久化以下状态：

1. 当前主线节点。
2. 当前段落或轮次索引。
3. 隐藏态度值。
4. 已读状态或历史记录摘要。
5. 基础设置项。

## 11. 错误处理要求

真实本地模型可能出现不可用、超时、流中断或空响应等情况。首版至少需要支持：

1. 模型不可用时的清晰提示。
2. 当前轮对话的重试。
3. 检索为空时的降级回答策略。
4. 流式输出中断时的恢复或中止提示。

## 12. 成功标准

若首版达到以下标准，则视为满足当前目标：

1. 玩家能在 Windows 桌面端完成一次不少于 5 分钟的单线体验。
2. 对话内容能明显体现《昆仑谣》相关文化主题，而不是空泛聊天。
3. 模型输出支持流式展示。
4. 两个选项每轮都能根据上下文动态生成。
5. 玩家可以感受到顺从与反驳带来的风格差异。
6. 游戏可以保存进度并恢复。

## 13. 当前明确不写死的内容

以下内容刻意保留到后续继续补充，不在本设计文档中写死：

1. 主线每个节点的具体剧情文案。
2. 用户后续整理的具体知识库原文内容。
3. 最终角色美术与背景美术素材本体。
4. 最终模型品牌、模型大小和具体部署方式。

这样处理的目的是先稳定产品结构、AI 约束和内容容器，再由用户持续填充文化内容和美术资源。
